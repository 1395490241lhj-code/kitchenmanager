# Inventory Mutation Coalescing (Phase 2B-4)

## Why coalescing exists

Before Phase 2B-4, every call to `InventorySyncAdapter.stageUpsert`/
`stageDelete` minted a fresh `mutationId` (`UUID()` default parameter) and
inserted a brand-new `PendingMutationRecord`. That was fine when the only
caller was `GuestMergeController.confirmMerge` — each merge candidate is
staged exactly once, right before a `runOnce`. Phase 2B-4 introduces a second
caller: ordinary CRUD, which can touch the *same* entity multiple times
before the next manual sync (edit a quantity twice, or create then delete
before ever syncing). Left unchanged, that would let an unbounded number of
pending mutations accumulate per entity, all eventually pushed together in
whatever order they happen to sort in — some of them describing states the
item never actually was in from the server's point of view.

`SwiftDataSyncPersistence.stageInventoryMutation(entityId:scope:operation:payloadData:now:)`
is the one place this is prevented: at most one *pending* `PendingMutation`
per `(entityType, entityId)` is ever kept. A second CRUD-originated call for
the same entity looks up that one existing pending mutation (if any) and
either updates it in place or replaces its intent — it never inserts a
second row.

## The rule table

| Existing pending | New CRUD op | Result |
| --- | --- | --- |
| none | create/update (`.upsert`) | insert a fresh `PendingMutation` — `mutationId = UUID()`, `baseVersion` = the entity's known `SyncMetadata.remoteVersion` (or `.zero` if none) |
| none | delete | insert a fresh delete `PendingMutation` — same baseVersion rule |
| `.upsert` (create or update) | `.upsert` | **replace the payload in place** — same `mutationId`, same `baseVersion`, only `payloadData`/`clientUpdatedAt` change. The version this mutation was staged against must never shift just because the local value changed again before the next sync. |
| `.upsert`, and the entity's `SyncMetadata.remoteVersion` is `nil` (i.e. this was a not-yet-sent *create*) | `.delete` | **cancel entirely** — the pending mutation *and* its `SyncMetadata` row are both deleted. The server never learns this entity ever existed; there is nothing to send. |
| `.upsert`, and the entity's `SyncMetadata.remoteVersion` is non-nil (i.e. this was an *update* on an already-synced item) | `.delete` | **merge into a single delete intent** — the existing `PendingMutation` row is rewritten in place (same `mutationId`) with `operation = .delete`, `payloadData = "{}"`, and `SyncMetadata.state` moves to `.pendingDelete`. The superseded update is never sent first. |
| `.delete` | `.upsert` | **refused** before reaching this method at all — `InventorySyncEligibility.evaluate` returns `.blockedByPendingDelete` for any update/create attempt on an entity with a pending delete, so the ordinary CRUD path never even calls `stageInventoryMutation` in this case. (The persistence method itself still has a defensive `throw` here as a fail-safe, never expected to fire.) |
| `.delete` | `.delete` | no-op — already staged, nothing further to do. |

Every branch above operates inside one `@ModelActor` transaction
(`SwiftDataSyncPersistence`'s own `ModelContext`), committed with a single
`try commit()` — there is no window where the mutation is written but the
metadata isn't, or vice versa.

## `mutationId` stability

A `PendingMutation`'s `mutationId` is set exactly once, at the moment it is
first inserted, and is never regenerated by any later coalescing call for
the same entity — this is the property `mutationId`-based idempotency (both
the local `@Attribute(.unique)` constraint and the server's per-`mutationId`
ledger) depends on. Verified directly: `GuestMergeTests.testCreateThenUpdateCoalescesIntoOneCreateMutationWithLatestPayload`
and `.testUpdateThenUpdateCoalescesIntoOneUpdateMutation` both assert the
`mutationId` is identical before and after a second local edit;
`.testUpdateThenDeleteCoalescesIntoASingleDeleteIntent` asserts the same
`mutationId` carries through an update-then-delete merge.

## What does *not* change

`SyncCoordinator.push`/`pull` are entirely unaware that coalescing exists —
they still just read whatever `PendingMutation`s are currently staged per
scope and send them. Retrying a mutation that the server has already applied
(a real network retry, not client-side coalescing) is unaffected: that is
the pre-existing server-side idempotency-ledger contract keyed on
`mutationId`, untouched by this phase.

## Where this is enforced, not just described

- `KitchenManager/Synchronization/SyncPersistence.swift` —
  `stageInventoryMutation(entityId:scope:operation:payloadData:now:)` on
  `SwiftDataSyncPersistence`.
- `KitchenManager/Synchronization/InventorySyncEligibility.swift` — the
  `.blockedByPendingDelete` case that prevents an update/create from ever
  reaching the coalescing method for an entity with a pending delete.
- `test/ios-native-guest-merge-phase2b1.test.mjs` — a Node semantic-guard
  test confirms all four `case (…, …):` coalescing branches exist in source,
  plus the "cancel entirely" and "merge into a single delete intent" comments
  that document *why* each branch exists (so the rule table above and the
  code can't silently drift apart).
